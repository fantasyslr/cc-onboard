# CC Onboard

> 新时代开源，不是开源代码而是开源对话。
>
> *code is cheap, show me the talk.*

---

我不会写代码。又菜又爱主导权——这是我给自己贴的标签。

听不懂就直接批准，走走看。这是我用 Claude Code 很长一段时间里的真实状态。

那段时间 AI 帮我做了很多事，但总有什么不对。它会乱改文件，会重复造轮子，会编造，会在我不需要解释的时候滔滔不绝。我以为是模型的问题。后来才发现：**是我从来没告诉它我是谁。**

每次对话它都从零开始猜我。它不知道我讨厌废话，不知道我想法驱动，不知道我要的永远是 MVP 而不是完整方案。它不知道这些，所以它给我一个平均值。

修这件事花了我很长时间——不是技术上的时间，是认知上的时间。你得先搞清楚自己是什么样的人，才能写出一份有用的配置。这不是填表，是自我剖析。

最后我写出了一份 `CLAUDE.md`。从那以后，AI 开始真正 work for me。

然后我意识到：这可能不只是我的问题。

---

## 这件事有多普遍

Claude Code 月活超过 **3000 万**，活跃开发者超过 **84,000**。SkillsMP 上已有 **7,000+** 个 skill。

但大多数人的 `CLAUDE.md` 长这样：

| 状态 | 比例 | 结果 |
|------|:---:|------|
| 没有 CLAUDE.md | 很多 | AI 每次从零猜你，你每次重新解释 |
| 抄了别人的 CLAUDE.md | 不少 | 那是别人的答案，问题是你的 |
| 写了但又长又杂 | 常见 | 10,000 token 的 CLAUDE.md 每条消息都加载，**50-70% 是无效内容** |
| 装了一堆 skill 和 hook | 我自己 | 互相打架，猛吃 token，不知道哪个在生效 |

第三行和第四行是我自己踩过的坑。说一下。

## 一个真实的踩坑故事

我的安装记录（不完全版）：

| 装了什么 | 为什么装 | 结果 |
|---------|---------|------|
| GSD（项目管理） | 看起来很专业 | plan→execute→verify 全生命周期，改一行字也给我画甘特图 |
| PUA（agent 加压） | GitHub 很火 | 有用，但全程开着变成背景噪音，token 猛吃 |
| CCG（多模型协作） | 火 | Codex + Gemini 协作听着很酷，实际上下文爆炸 |
| ECC（增强版 Claude Code） | 更火 | 跟其他 skill 权力冲突 |
| DeerFlow（字节的搜索 agent） | 字节出的 | 又加一层 context |
| 各种 memory 插件 | 总能用上吧 | 互相覆盖，不知道谁在生效 |
| Superpowers（skill 路由） | 管理上面那堆 | 又多一层要加载的规则 |

**谁火装谁**——又菜又爱主导权的人干出来的事。结果：

- **Token 爆炸** — 多套 skill 同时加载，一个简单问题先烧 30% 的 context 在读规则。臃肿的 CLAUDE.md 每条消息都带着，**50-70% 是无效内容**
- **互相打架** — GSD 说「按 plan 走」，PUA 说「换方案」，Superpowers 说「先查 skill」，谁听谁的？
- **不知道哪个在生效** — 我不写代码，我看不出来哪个 skill 在影响 AI 的行为，只知道「感觉不对」
- **简单任务开销太大** — 改一行配置，AI 给我画 Sprint Banner + KPI 卡 + 压力面板

最后我花了一整轮对话，让 AI 追问我「你到底需要什么」——这个追问过程本身把我的痛点暴露了出来。我才真正搞清楚：不是 skill 不好，是**没有人帮我选对的 skill**。

后来我砍掉了大部分，只留真正需要的。Token 消耗直接降了 **60%+**。

**这就是为什么 CC Onboard 要集成 skill 推荐** —— SkillsMP 上有 **7,000+** 个 skill，不是给你一张目录让你自己挑。是先搞清楚你是谁，然后只装你需要的。谁火装谁，这条路我替你走过了，别再走一遍。

## 你可能是这样的

不是「你是不是这样的人」——是你可能正在经历这些事：

| 你的现状 | 你的感受 | 根因 |
|---------|---------|------|
| AI 用了不少，但每次打开都要重新解释自己 | 烦，但不知道怎么办 | 没有 CLAUDE.md，或者有但不是你的 |
| 看了别人的配置分享，抄了，感觉一般 | 差一口气，说不清差在哪 | 抄的是答案，你的问题没被问过 |
| 装了 Cursor / Copilot / Claude Code，用了一阵 | 「还行」，然后就这样了 | 没进入 agent 模式，还在一问一答 |
| 装了各种 skill 和 hook | 不知道哪个有用，不敢删 | 没人根据你的需求做过匹配 |
| AI 帮你做事，但它乱改文件、编造、废话 | 想骂它但觉得可能是自己的问题 | 确实是你的问题——你没告诉它什么不能做 |
| 知道应该认真配置，但不知道从哪下手 | 搜了教程，越看越迷 | 你需要的不是教程，是有人问你对的问题 |

**如果以上命中了任何一条——这个 skill 就是为你做的。**

你不需要会写代码。你不需要懂什么是 hook、什么是 context window、什么是 token。你只需要能回答一些关于自己的问题——「AI 最让你抓狂的一次是什么情况」「你更希望 AI 主动还是等指令」——然后它帮你把答案翻译成机器能读懂的语言。

## 它做什么

**一句话：8 个问题认识你，然后帮你配好一切。**

### 第一步：采访你

8-10 个问题。不是选择题，不是量表，不是 60 道考试卷。

是具体的场景追问——「AI 最让你抓狂的一次是什么情况」，而不是「你喜欢简洁还是详细的回答风格」。

为什么不做选择题？因为人面对选项会乱选、跳过、选个「差不多」的。只有让你用自己的话描述一个具体场景，真实偏好才会浮出来。我们也试过选择题分叉（根据上一题生成下一题的选项），太重了，而且本质问题没变。

### 第二步：生成你的 CLAUDE.md

基于你说的话，用你的语气，写出你的个人 AI 契约。

同时判断你是「先斩后奏」型还是「每步确认」型——如果你需要安全锁（AI 改文件前先问你），自动配上；如果你更喜欢它直接干，就不加。

### 第三步：推荐并安装 skill

**这是我踩坑之后最想做的功能。**

根据你的使用场景匹配工具。你说「我主要用 AI 分析数据」→ 它找数据相关的 skill。你说「我需要它帮我管项目」→ 它推荐项目管理 skill。不是给你 7,000 个 skill 的目录让你自己挑。

### 三档深度，你来选

| 档位 | 包含什么 | 适合谁 | 耗时 |
|------|---------|-------|------|
| **Quick Start** | 只有 CLAUDE.md | 想试试，不想折腾 | 5 分钟 |
| **Balanced** | CLAUDE.md + 几个 skill + 可选 hooks | 日常主力配置 | 10 分钟 |
| **Full Setup** | 全套：skill、hooks、memory | 重度用户 | 15 分钟 |

| **alignment-interview-cc** | 60 题深度访谈 | 像考试卷，大多数人做到一半关掉 |

| **gsd:profile-user** | 被动分析历史对话 | 不主动介入，你得先有足够的对话历史 |

| **CC Onboard** | 8-10 个追问 + 自动配置 | **帮你发现你是谁，而不是问你是谁** |

问卷给你选项，你选一个最接近的。追问逼你描述场景，真正的自己才出来。

## 为什么 CLAUDE.md 需要持续维护

你的第一份 CLAUDE.md 一定是不完整的。

用一周，你会发现「哦，我还讨厌它做 X」或者「其实我有时候希望它解释一下」。这很正常——你在和一个系统磨合，第一轮不可能完全准确。

`/onboard review` 是为这个设计的。3-4 个针对性问题，提出修改建议，不覆盖现有内容。把它当季度复盘，而不是装完扔掉。

理想状态是 AI 在后台默默观察你的使用习惯，自动帮你优化配置。但这件事容易做歪——它一旦自动改你的配置，你发现了，第一反应是关掉它。所以我们选了最土但最靠谱的方式：**你主动来跑，它被动响应。** 又菜又爱主导权嘛——那主导权就给你，一直给你。

### CLAUDE.md 不止一层

| 层级 | 路径 | 管什么 |
|------|------|-------|
| **你这个人** | `~/.claude/CLAUDE.md` | 你跟 AI 的个人契约，跨项目生效。CC Onboard 生成的就是这个 |
| **这台机器** | `~/CLAUDE.md` | Claude Code 在这台机器上怎么干活 |
| **具体项目** | 项目根目录的 `CLAUDE.md` | 项目特定的规则和上下文 |

刚开始只需要管第一层。

## 安装

```bash
npx skills add slr/cc-onboard -g -y
```

## 使用

第一次：
```
/onboard
```

用过一段时间后的复盘：
```
/onboard review
```

所有数据留在本地。没有外发。

## 关于这个项目

这个 skill 本身是对话的产物。

先是一轮 harness 设计的讨论——怎么把三套不同的 AI 工具组合成一个适合自己的工作流。在那个过程中，AI 用追问帮我发现了自己的使用偏好和痛点。然后我意识到：这个「追问→发现自己」的过程本身就是一个产品。

设计对话开源在 `archive/`。你拿走代码，能复制一个工具。你拿走对话，能看到这个工具是怎么被想出来的。

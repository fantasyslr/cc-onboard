# CC Onboard

> 新时代开源，不是开源代码，而是开源对话。
>
> *Code is cheap, show me the talk.*

---

我不会写代码。

我给自己贴过一个很准确的标签：**又菜又爱主导权。**

用 Claude Code 很长一段时间，我的工作方式都是这样：听不懂，先批准，看看会发生什么。AI 的确帮我做了很多事，但总有一种挥之不去的别扭感。它会乱改文件，会重复造轮子，会编造，会在我只想要结果的时候滔滔不绝。我一开始以为是模型的问题，后来才意识到：**不是它不够聪明，是我从来没告诉它我是谁。**

每次对话，AI 都在从零开始猜你。它不知道你讨厌废话，不知道你是想法驱动还是流程驱动，不知道你更想要一个能陪你拆想法的搭档，还是一个少说多做的执行者。它只能给你一个平均值。

而写一份真正有用的 `CLAUDE.md`，难的从来不是语法，是自我认识。你得先讲清楚：我最讨厌 AI 做什么？我希望它什么时候主动，什么时候闭嘴？什么能碰，什么绝对不能碰？大多数人卡在这一步，不是不会配，而是从来没人问对问题。

`CC Onboard` 就是为这件事做的。

它不是一份配置模板，也不是一张调查问卷。它是一场对话：用 5 个具体问题，把你的工作方式、偏好、耐受度和边界问出来，然后帮你生成一份 AI 真的会遵守的个人契约。

---

## 它是给谁用的

这个 skill 写给这样一类人：

- 不会写代码，但对结果、节奏和品味有很强要求。
- 已经用了很多 AI，但每次打开新对话，还是得重新解释自己是谁。
- 抄过别人的 `CLAUDE.md`，能用，但总觉得差一口气。
- 装过不少 skill 和 hook，最后发现不是更强，而是更乱、更贵、更慢。
- 想真正进入 agent 时代，但不知道第一步该从哪开始。

我对这类人的描述一直很直白：**不会写代码，但坚持要掌舵。**

这不是缺点。这恰恰意味着你对 AI 的要求不是“给我一个答案”，而是“按我的方式和我一起干活”。

---

## 它到底做什么

`CC Onboard` 做四件事：

### 1. 先采访你，不先教育你

它会问你 5 个问题。不是量表，不是考试，更不是让你在一堆模糊选项里选”最接近的一项”。

它问的是具体情境，比如：

- AI 最让你烦的一次是什么？
- 你更怕它擅自改东西，还是更怕它太被动、推一步走一步？
- 你希望它像搭档、助理、老师，还是纯执行器？

真正的偏好不会出现在抽象题里，只会出现在你讲一个具体场景的时候。

### 2. 帮你写出你的 `CLAUDE.md`

基于你的回答，它会生成一份放在 `~/.claude/CLAUDE.md` 的个人契约。

这份文件不是“最佳实践合集”，而是你和 AI 的工作协议。它会告诉 Claude：

- 应该怎么跟你说话
- 什么是你的高压线
- 什么时候该主动，什么时候该确认
- 你要的是过程，还是结果

写得好的 `CLAUDE.md` 会让 AI 从“通用助手”变成“真的懂你的人”。

### 3. 只在你需要时，给你加安全锁

有些人希望 AI 直接动手，做完再看结果。有些人最怕的就是“它先改了再说”。

`CC Onboard` 不会把所有人都当成一种用户。它会根据你的回答判断要不要给文件操作加 safety hooks，把“别乱来”真正落成系统行为，而不是一句空话。

### 4. 帮你挑 skill，而不是让你逛技能超市

我踩过一个很典型的坑：谁火装谁。

各种增强工具，装的时候都觉得自己在变专业，最后的结果往往是：规则互相打架，AI 还没开始干活就先花一大堆精力在读规则上，简单任务也被搞成重型流程。

所以这个 skill 不会把你丢到一个巨大的工具货架前让你自己挑。它会先搞清楚你要 AI 帮你做什么，再推荐真的匹配的那几个——推荐之前还会检查这个工具靠不靠谱（有多少人在用、是谁做的）。

## 为什么不用问卷，而要用追问

这是这个项目最核心的设计判断。

我们当然想过更“系统”的做法，比如大量选择题，或者根据上一题动态生成分叉选项。最后都放弃了。

原因很简单：

- 选择题很快，但人会乱选、跳过、选个差不多。
- 分叉问卷很聪明，但很重，逻辑成本高，用户体验并不好。
- 只有当你讲出一个真实场景，真正的偏好才会自己浮出来。

所以 CC Onboard 的原则只有一句话：

**帮你发现你是谁，而不是问你是谁。**

---

## 为什么它不是一次性工具

理想状态当然很诱人：AI 在后台默默观察你怎么用它，然后自动帮你把配置越调越准。

但这件事一旦做不好，就会立刻击中这类用户最核心的心理边界：**主导权必须在我手里。**

它偷偷改你的配置，哪怕是好意，你第一反应也很可能是关掉它。

所以这个 skill 选了一条更笨、但更可靠的路：

- 先做 onboarding，这是必做的第一步。
- 不做后台接管，而是做成可重跑。
- 当你觉得“它最近又有点不对味了”，就运行一次 `/onboard review`。

也就是说，我们放弃了“自动观察优化”，换来了“随时可复盘、始终由你拍板”。

---

## 三档深度，你自己定

- **Quick Start**: 只生成 `CLAUDE.md`。适合想先快速进入状态的人。
- **Balanced**: `CLAUDE.md` + 核心 skill + 可选安全锁。适合大多数人的日常主力配置。
- **Full Setup**: 再往上加更完整的 skill、hooks 和记忆能力。适合重度使用者。

不是所有人都该一步到位。真正好的配置，不是越满越强，而是越贴合你越强。

---

## 你的 AI 契约其实有三层

Claude Code 的个人配置不止一份，而是分了三层：

- **你这个人的契约**（`~/.claude/CLAUDE.md`）：不管你在做什么项目，这份契约都生效。CC Onboard 生成的就是这个。
- **某台机器或某个项目的规则**：比如「这个项目用 Python」「部署在阿里云」。
- **更细粒度的局部规则**：针对项目里某个特定模块。

刚开始只需要管第一层。等你用熟了，自然会知道什么时候需要后面两层。

---

## 这和别的 onboarding 有什么不一样

我看过几类现成方案：

- 有的像考试卷，问题很多，用户做到一半就关掉。
- 有的把 onboarding 绑进一整套课程或方法论里，目标不够纯。
- 有的是被动分析你的历史对话，不会主动把你带进这个过程。

`CC Onboard` 的差别不在于“题目更少”或者“语气更温柔”，而在于它把真正有价值的那一步产品化了：

**追问你，直到你的真实工作方式自己显形。**

## 项目状态

| 模块 | 状态 | 说明 |
|------|:---:|------|
| Onboarding 流程 | ✅ 可用 | 5 问采访 + 自动分类 + 配置生成 |
| Review 模式 | ✅ 可用 | 复盘校准，不覆盖原有配置 |
| 翻译规则 | ✅ 内置 | 回答→配置的映射表，半结构化 |
| 示例 | ✅ 完整 | `examples/` 目录：两种用户的完整 5 问对话 + 生成结果 + 翻译逻辑 |
| 安全锁自动配置 | ✅ 可用 | 根据用户偏好决定是否添加 |
| 工具推荐 | ✅ 可用 | 集成质量验证（安装量、来源信誉），推荐前先检查靠不靠谱 |
| 输出稳定性验证 | 🔜 计划中 | 不同类型用户的对比测试 |
| 录屏 / Demo | 🔜 计划中 | 完整 onboarding 流程的录屏演示 |

**现在就可以用。** 装上就能 `/onboard`。

翻译规则依赖 AI 的理解力而非硬编码逻辑——这意味着对同一个回答，不同时候可能生成略有差异的配置。如果生成的契约不够准，跑一次 `/onboard review` 就能校准。

## 安装

```bash
npx skills add fantasyslr/cc-onboard -g -y
```

## 使用

第一次使用：

```bash
/onboard
```

当你已经用了一阵，想复盘和校准：

```bash
/onboard review
```

没有 `CLAUDE.md` 时，会进入完整 onboarding。

已经有 `CLAUDE.md` 时，会进入 review 模式，只提 3-4 个针对性问题，给你修改建议，不会直接覆盖。

所有数据都留在本地，不会离开你的机器。

## 这个项目怎么来的

它不是先有 PRD，再有实现。

它先来自两轮对话。

第一轮，我在想怎么把几套 AI 工作流工具揉成一个适合自己的 harness。聊着聊着，AI 通过追问把我的偏好、痛点和边界一点点逼了出来。第二轮，那个更重要的结论才浮出来：**真正值得产品化的，不是某个神奇配置，而是“追问你，帮你发现自己如何和 AI 合作”的过程。**

所以这个仓库不只开源 skill，也尽量开源这个 skill 是怎么被想出来的。你可以在 `archive/` 里看到它从脑暴到收敛的痕迹。

拿走代码，你可以复制一个工具。

拿走对话，你可以看到一个工具是怎么长出来的。

这才是我理解的“开源对话”。
